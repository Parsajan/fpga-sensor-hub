UART TX — Sending byte 0x41 ('A')

Frame: Start bit + 8 data bits (LSB first) + Stop bit
What to look for
✓TX_PIN = '1' at idle, before TX_START fires
✓TX_BUSY goes high immediately after TX_START
✓First fall of TX_PIN to '0' = start bit, lasting exactly 8.68 µs
✓For 0x41: bit order is 1,0,0,0,0,0,1,0 (LSB first)
✓Last bit is the stop bit: TX_PIN = '1'
✓TX_DONE pulses high for exactly one clock cycle
✗If TX_PIN glitches mid-bit → check the baud generator
✗If TX_BUSY drops before the stop bit → FSM bug
Expected Transcript output

# --- test 1: send 0x41 (A) ---
# [PASS] tx_busy=1 confirmed
# [MON] received: 0x41
# [PASS] test 1 done: sent 0x41
# --- test 2: send 5 bytes ---
# [PASS] byte 0 = 0x41 sent
# [PASS] byte 1 = 0x42 sent
...
# === ALL uart_tx TESTS PASSED ===

UART RX — Receiving and mid-bit sampling

Key idea: receiver samples the MIDDLE of each bit, not the edge
What to look for
✓After RX_PIN falls, STATE moves to START
✓Sampling happens exactly at HALF_BAUD_DIV = 434 clocks (mid-bit)
✓Green dots = bit sampled as '1', red dots = bit sampled as '0'
✓After 8 bits, RX_VALID pulses high and RX_DATA = 0x55
✓RX_SYNC2 lags RX_PIN by 2 clock cycles (double-flop synchronizer)
!Glitch test: a pulse shorter than 434 clocks should NOT move state out of IDLE
✗If RX_DATA is wrong → check baud divider or sampling point
Expected Transcript output
# --- test 1: send 0x55 ---
# [PASS] test 1: received 0x55
# [PASS] test 2: received 0x00
# [PASS] test 3: received 0xFF
# --- test 4: glitch (50ns, must be ignored) ---
# [PASS] test 4: glitch ignored correctly
# [PASS] received 0x48 (H)
# === ALL uart_rx TESTS PASSED ===

SPI Master — TX=0xD0, RX=0x60 (BME280 CHIP_ID)

Mode 0: CPOL=0, CPHA=0 — sample on rising edge, shift on falling edge
What to look for
✓CS_N goes low before SCLK starts toggling
✓SCLK idles at '0' (confirms Mode 0)
✓Exactly 8 full SCLK pulses for 8 data bits
✓MOSI = 1,1,0,1,0,0,0,0 = 0xD0 (MSB first)
✓MISO = 0,1,1,0,0,0,0,0 = 0x60 returned by slave model
✓After the last clock, CS_N goes high and DONE pulses for one cycle
✗If CS_N rises at the same time as the last SCLK edge → timing bug
Expected Transcript output
# [PASS] initial state: CS_N='1', SCLK='0' (Mode 0 ok)
# --- test 1: send 0xD0, expect RX=0x60 ---
# [SLAVE] received: 0xD0 sent: 0x60
# [PASS] test 1: RX = 0x60 (CHIP_ID ok)
# [PASS] busy='1' during transfer
# === ALL spi_master TESTS PASSED ===

PWM Generator — duty cycle comparison

Each row = one full PWM period (1000 clocks at sim CLK_FREQ=1MHz)
What to look for
✓duty=0x00: output stays '0' the entire period, never toggles
✓duty=0xFF: output stays '1' the entire period
✓duty=0x80: exactly half the period HIGH, half LOW
✓The measure process in the Transcript should report percent=50%
✓After mid-cycle reset: pwm_out = '0' and the counter restarts from zero
!During dynamic duty changes, one transitional period may look slightly off — that's expected
Expected Transcript output
# --- test 1: duty=0x00 ---
# [PASS] test 1: duty=0, output='0'
# [PASS] test 2: duty=255, output='1'
# [MEASURE] duty=128 high_clks=500/1000 percent=50%
# [MEASURE] duty=64 high_clks=250/1000 percent=25%
# [PASS] test 6: reset ok
# === ALL pwm_gen TESTS PASSED ===
